# PC용 Qorvo 연계 — 360° 레이더 테스트 콘솔 요구사항 정의서

> **한 줄 정의:** DWM3001CDK 보드에서 나오는 **거리·각도 데이터가 정상적으로 측정되는지 빠르게 확인**하고, 안 나올 경우 **로그로 원인을 진단**하기 위한 PC용 시리얼 테스트 콘솔(Flet/Python).
>
> **이 문서의 성격:** 상용 기능 명세가 아니라 **초도 기능 검증(Bring-up Test)**용 최소 요구사항. "많은 기능"보다 "**빠른 기본 동작 확인**"이 최우선.
>
> **사용 방식:** 이 문서는 **AI 코딩 도구(Cursor / Claude 등)에 그대로 물려 앱을 생성**하기 위한 빌드 스펙입니다. 그래서 기술 스택·데이터 포맷·모듈 구조·완료 조건을 **모호함 없이 고정**했습니다. 바로 쓰려면 **9장(구현 가이드)과 맨 끝의 '복붙용 빌드 프롬프트'**부터 보세요.
>
> 작성 기준일: 2026-06-24 · 버전: v0.2 (Vibe Coding용)

## 0. 문서 개요

### 0.1 목적
- DWM3001CDK 보드를 PC에 연결했을 때, **거리(cm)와 각도(°)가 실제로 측정되어 들어오는지**를 육안으로 즉시 확인한다.
- 값이 안 들어오거나 이상할 때, **무엇 때문인지(연결·포맷·신호)** 로그로 추적할 수 있게 한다.

### 0.2 범위 (Scope)

| 구분 | 내용 |
|---|---|
| **In-Scope (포함)** | 시리얼 연결, 데이터 파싱, 360° 레이더 표시, 실시간 수치 표시, 로그 콘솔·저장, 연결상태 표시, **BLE OOB/세션 상태 관찰·표시**(보드가 UART로 보고하는 세션 라이프사이클) |
| **Out-of-Scope (제외)** | 결제 승인/서버 연동, Zone별 결제 판정 로직, 모바일 앱 자체, 사용자 계정, 다중 보드 동시 관제, 칼만필터 등 고급 보정(※ 초도 테스트에서는 **원시값 우선 확인**), **PC가 BLE 무선을 직접 수행하는 것**(OOB는 보드↔폰 사이에서 일어나며 PC는 USB-UART로 관찰만) |

> **테스트 토폴로지:** 보드(DWM3001CDK) ↔ 스마트폰. 스마트폰 앱(Android Jetpack UWB / iOS NI)이 BLE OOB를 수행하고, 보드가 상대측이 된다. **PC 콘솔은 보드의 UART 로그로 세션 진행 상태를 관찰**한다. (폰 측 OOB 자체 디버깅은 별도 모바일 테스트의 몫)

### 0.3 용어

| 용어 | 설명 |
|---|---|
| **보드** | Qorvo DWM3001CDK. 거리·각도를 측정해 PC로 시리얼 전송 |
| **PC 앱 / 테스트 콘솔** | 본 문서가 정의하는 Flet 기반 PC 프로그램 |
| **시리얼(UART)** | 보드와 PC가 글자 데이터를 주고받는 통신 (USB-Serial) |
| **거리(Distance)** | 보드~태그 간 직선 거리, 단위 cm |
| **각도(Angle, AoA)** | 보드 정면 기준 방위. 0°=정면, 음수=좌, 양수=우 |
| **레이더 뷰** | 거리·각도를 원형(폴라) 좌표에 점으로 찍는 화면 |

## 1. 개요

### 1.1 시스템 구성

```
 ┌───────────────────────┐    USB / 시리얼(UART)   ┌────────────────────────────┐
 │  DWM3001CDK 보드       │  ───────────────────▶   │  PC 테스트 콘솔 (Flet)      │
 │  · 거리/각도 측정       │   "DIST:85,ANGLE:-12"   │  · 파싱 → 레이더 표시        │
 │  · 펌웨어가 값 전송     │                         │  · 수치 패널 + 로그 콘솔     │
 └───────────────────────┘                         └────────────────────────────┘
         (측정 담당)                                       (표시·진단 담당)
```

- **보드(펌웨어)**가 측정·연산을 담당하고, **PC 앱**은 받은 값을 **표시·기록**만 한다. (역할 분리)
- 통신은 단방향 수신이 기본(보드 → PC). 필요 시 시작/정지 명령 전송은 선택 기능.

### 1.2 이 앱이 하지 않는 것 (명확화)
- 거리값을 가공·보정하지 않는다 (원시값 그대로 표시가 기본 — 진단 목적).
- 결제 여부를 판단하지 않는다.
- 펌웨어를 수정/플래싱하지 않는다 (그건 SES의 역할).
- **BLE 무선을 직접 다루지 않는다.** OOB 핸드셰이크는 보드↔폰이 하고, PC는 보드가 UART로 보고하는 상태만 표시.

### 1.3 테스트 2단계 & BLE OOB 관찰 (중요)

UWB ranging 값은 **BLE OOB로 세션이 맺어진 뒤에야** 나온다. 따라서 거리값이 안 나올 때 원인을 갈라보려면 **세션 단계**를 함께 봐야 한다. 테스트는 두 단계로 진행한다.

| 단계 | 내용 | 이 앱의 역할 |
|---|---|---|
| **A. 정적 파라미터 모드 (OOB 우회)** | 보드·폰에 세션값을 하드코딩해 OOB를 건너뛰고 **순수 UWB ranging + 거리/각도 + 표시**만 검증 | 거리/각도 표시 (3장 기본 기능) |
| **B. OOB 포함 전체 흐름** | BLE 깨우기 → GATT 연결 → 파라미터·키 교환 → UWB 세션 점화까지 실제 수행 | 보드가 보고하는 **세션 상태(T-0~T-3)를 타임라인으로 관찰** |

> **세션 상태 = 상위 가이드의 T-0~T-3 핸드셰이크.** SLEEP → BLE 광고 → BLE 연결 → OOB(파라미터·키 교환) 완료 → RANGING 시작. 어느 단계에서 멈췄는지 보이면, "거리값 없음"의 원인(BLE 미연결 vs OOB 실패 vs PHY 문제)을 즉시 가를 수 있다.

## 2. 사용자 시나리오

**페르소나:** 펌웨어·앱 개발자(테스터) 1인. 보드를 책상에 두고 동작을 확인한다.

### 2.1 정상 시나리오 (Happy Path)
1. 보드를 USB로 PC에 연결한다.
2. 앱을 실행하면 사용 가능한 **COM 포트 목록**이 보인다.
3. 포트를 고르고 **[연결]**을 누른다.
4. 화면 중앙 레이더에 **점이 나타나고**, 손/태그를 움직이면 점이 거리·각도에 맞게 움직인다.
5. 우측 수치 패널에 **거리 85cm / 각도 −12°** 같은 값이 실시간 갱신된다.
6. 하단 로그에 수신 원시 데이터가 흐른다. → **"측정 정상" 확인 완료.**

### 2.2 비정상 시나리오 (진단 흐름) ★이 앱의 핵심 가치
- **점이 안 보임** → 로그를 본다 → 데이터 자체가 없으면 "수신 0건", 깨졌으면 "파싱 실패: 원문 표시" 로 **원인 구분**.
- **값이 튐** → 로그에 원시값이 그대로 찍히므로, 튀는 게 보드 출력인지 표시 문제인지 판별.
- **연결이 끊김** → 상태 표시가 '연결됨→끊김'으로 바뀌고 로그에 시각과 함께 기록.

## 3. 기능 요구사항 (FR)

각 요구사항은 우선순위 표기: **[必]** 필수 · **[選]** 선택.

| ID | 기능 | 우선 | 상세 |
|---|---|---|---|
| FR-1 | **포트 선택·연결 관리** | 必 | 사용 가능한 COM 포트 자동 탐색, Baudrate 선택(기본 115200), 연결/해제 버튼 |
| FR-2 | **데이터 수신·파싱** | 必 | 시리얼 라인을 읽어 거리·각도 값 추출. 정의된 포맷(5장)과 다르면 '파싱 실패'로 분류 |
| FR-3 | **360° 레이더 표시** | 必 | 거리=중심에서의 반지름, 각도=방위로 점 1개 표시. 동심원(거리 눈금)·각도 눈금 포함 |
| FR-4 | **실시간 수치 패널** | 必 | 현재 거리(cm)·각도(°)·수신율(건/초)·마지막 수신 시각 표시 |
| FR-5 | **로그 콘솔** | 必 | 수신 원시 라인·파싱 결과·에러를 시각(timestamp)과 함께 스크롤 표시 |
| FR-6 | **로그 파일 저장** | 選 | 세션 로그를 .txt/.csv로 저장 (사후 분석용) |
| FR-7 | **연결 상태 표시** | 必 | '연결됨/끊김/수신없음' 상태를 색상으로 항상 표시 |
| FR-8 | **시작/정지 명령 전송** | 選 | 보드에 ranging 시작·정지 명령 전송 (펌웨어가 지원할 경우) |
| FR-9 | **표시 일시정지/초기화** | 選 | 화면 멈춤·로그 클리어 버튼 |
| FR-10 | **세션/OOB 상태 표시** | 必 | `STATE:` 메시지를 파싱해 T-0~T-3 단계 타임라인과 현재 상태등 표시. `ERR` 시 사유를 강조 표시 |

## 4. 비기능 요구사항 (NFR)

초도 테스트 도구이므로 **가볍게**, 단 측정 신뢰를 해치지 않는 선에서 수치를 명시한다.

| ID | 항목 | 기준 |
|---|---|---|
| NFR-1 | **표시 지연** | 데이터 수신 → 화면 반영까지 **100ms 이내** (체감 실시간) |
| NFR-2 | **갱신율** | 레이더·수치 패널 **10fps 이상** 부드럽게 갱신 |
| NFR-3 | **무응답 감지** | **2초** 이상 데이터 없으면 '수신없음'으로 자동 표시 |
| NFR-4 | **자동 재연결** | 케이블 재연결 시 수동 재선택 없이 복구(선택), 최소한 명확한 끊김 안내(필수) |
| NFR-5 | **안정성** | 깨진 데이터가 들어와도 앱이 죽지 않고 로그에 남긴다 (파싱 실패 무중단) |
| NFR-6 | **실행 환경** | Windows 11, Python 3.x, 단일 실행(추가 서버 불필요) |
| NFR-7 | **구조** | 시리얼 수신은 **백그라운드 스레드**, UI는 메인 스레드 — 화면 버벅임 방지 |

## 5. 시리얼(UART) 통신 인터페이스 정의 ★보드 팀 합의 필요

> 이 장은 **보드(펌웨어)와 PC 앱이 반드시 같은 약속을 쓰도록** 못 박는 부분입니다. 펌웨어 출력 포맷이 바뀌면 여기부터 갱신합니다.

### 5.1 물리 계층

| 항목 | 값 |
|---|---|
| 인터페이스 | USB CDC / Virtual COM Port |
| Baudrate | **115200** (기본, 앱에서 변경 가능) |
| Data/Parity/Stop | 8 / None / 1 |
| 줄 구분 | `\n` (LF) 단위로 1개 측정 |

### 5.2 메시지 포맷 (제안 — 보드 팀과 확정 필요)

펌웨어가 한 줄에 한 측정을 보내는 **사람이 읽을 수 있는 형식**을 기본으로 한다.

```
DIST:85,ANGLE:-12
```

| 필드 | 의미 | 단위/범위 |
|---|---|---|
| `DIST` | 거리 | cm, 0 ~ 약 5000 |
| `ANGLE` | 각도(AoA) | °, -90 ~ +90 (0=정면) |

- **확장 여지:** 신호세기(`RSSI`), 측정 신뢰도(`Q`), 타임스탬프 추가 가능 → 추가 필드는 **무시 가능하게** 파싱(앞으로 호환).
- **거리만 나오고 각도 미지원 단계**도 있을 수 있으므로, `ANGLE` 없으면 **거리만 표시**하고 각도는 'N/A'로 처리.

### 5.3 파싱 규칙
- 정의된 키(`DIST`,`ANGLE`)를 찾아 숫자만 추출. 키 순서 무관.
- 라인이 비었거나 키가 없으면 **파싱 실패**로 분류하되 원문을 로그에 남긴다.
- 숫자 변환 실패·범위 초과 값은 '이상치'로 로그 표기(표시는 하되 색상 경고).

### 5.4 세션/OOB 상태 메시지 (BLE OOB 관찰용) ★보드 팀 합의 필요

보드 펌웨어가 BLE OOB·UWB 세션 라이프사이클을 **한 줄 상태 메시지**로 UART에 함께 흘려보낸다. 거리 데이터와 같은 시리얼로 섞여 들어오며, 앱은 `STATE:` 접두로 구분한다.

```
STATE:BLE_ADV
STATE:BLE_CONN
STATE:OOB_DONE
STATE:RANGING
STATE:ERR,REASON:OOB_TIMEOUT
```

| 상태 값 | 의미 | 대응 단계 |
|---|---|---|
| `SLEEP` | UWB OFF, 대기 | T-0 |
| `BLE_ADV` | BLE 광고(폰 깨우기) 중 | T-1 |
| `BLE_CONN` | 폰과 BLE(GATT) 연결됨 | T-2 |
| `OOB_DONE` | UWB 파라미터·보안키 교환 완료 | T-2 완료 |
| `RANGING` | UWB 세션 점화, 거리 측정 시작 | T-3 |
| `ERR,REASON:<사유>` | 단계 실패 (예: `BLE_FAIL`, `OOB_TIMEOUT`, `KEY_MISMATCH`) | 임의 단계 |

- 미지원 펌웨어(상태 미전송)면 앱은 세션 타임라인을 'N/A'로 두고 거리 표시는 정상 동작.
- `REASON` 텍스트는 자유 문자열 — 앱은 그대로 로그·툴팁에 노출.

### 5.5 (선택) PC → 보드 명령
펌웨어가 지원하는 경우에 한해, 텍스트 명령 전송:

| 명령 | 동작 |
|---|---|
| `START\n` | ranging 시작 |
| `STOP\n` | ranging 정지 |

## 6. 화면 설계 (UI)

### 6.1 레이아웃 개요

```
┌──────────────────────────────────────────────────────────┐
│ [연결 바]  포트:[COM3 ▼]  속도:[115200 ▼]  ●연결됨   [연결/해제] │
├──────────────────────────────────────────────────────────┤
│ [세션 타임라인]  ①SLEEP ─▶ ②BLE_ADV ─▶ ③BLE_CONN ─▶ ④OOB_DONE ─▶ ⑤RANGING │
│                  ●완료    ●완료      ●완료      ●진행중     ○대기         │
├───────────────────────────────┬──────────────────────────┤
│                               │  [수치 패널]               │
│        360° 레이더 뷰          │   거리   :  85 cm          │
│      (동심원 + 각도 눈금)       │   각도   : -12 °           │
│            ● 태그              │   수신율 : 12 건/초         │
│                               │   최종수신: 12:03:41        │
│                               │   세션   : ● OOB_DONE      │
│                               │   상태   : ● 정상           │
├───────────────────────────────┴──────────────────────────┤
│ [로그 콘솔]                                  [저장][지우기]  │
│ 12:03:41.210  RX  DIST:85,ANGLE:-12        ✔ 파싱 OK        │
│ 12:03:41.260  RX  DI?T:8x,ANGL...          ✘ 파싱 실패       │
└──────────────────────────────────────────────────────────┘
```

### 6.2 레이더 좌표계 규약 (반드시 통일)
- **중심** = 보드(앵커) 위치.
- **반지름** = 거리. 동심원 눈금을 cm로 라벨(예: 50 / 100 / 200 / 300cm). 최대 표시 거리는 설정값.
- **각도** = 0°를 **정면(위쪽 12시 방향)**, 좌측 음수(−), 우측 양수(+). 표시 범위 −90°~+90°.
- 좌표 변환: 화면 x = r·sin(θ), y = -r·cos(θ) (정면이 위로 가도록). ※ 구현 시 좌우 반전 주의.
- 데이터 없을 땐 점을 숨기고 레이더 배경만 유지.

### 6.3 컴포넌트별 명세

| 컴포넌트 | 표시/동작 |
|---|---|
| 연결 바 | 포트 드롭다운(자동 갱신), Baudrate, 연결 상태 LED(녹/회/적), 연결 토글 |
| 세션 타임라인 | T-0~T-3(SLEEP→BLE_ADV→BLE_CONN→OOB_DONE→RANGING) 5단계를 가로로 표시. 완료=녹, 진행중=점멸, 대기=회, 실패=적+사유. **거리값이 없을 때 어디서 막혔는지 보여주는 핵심 위젯** |
| 레이더 뷰 | 동심원·각도선·태그 점. 점 옆에 거리/각도 숫자 라벨(선택) |
| 수치 패널 | 거리·각도·수신율·최종수신시각·상태. 값 갱신 시 강조 |
| 로그 콘솔 | timestamp + 방향(RX/TX) + 원문 + 파싱결과(색상). 자동 스크롤, 일시정지 |
| 상태 표시 | 정상(녹)·수신없음(회)·끊김/에러(적) |

### 6.4 상태별 화면 동작

| 상태 | 화면 |
|---|---|
| 미연결 | 레이더 비활성(흐림), 수치 '—', 상태 회색 |
| 연결·수신중 | 점 표시, 수치 갱신, 상태 녹색 |
| 수신없음(2초+) | 점 사라짐, 상태 회색 '수신없음', 로그에 경고 |
| 끊김/에러 | 상태 적색, 마지막 값 고정 표시, 로그에 원인 기록 |

## 7. 예외 처리 & 진단 (Corner Cases)

이 앱의 존재 이유. "왜 안 나오는지"를 사람이 판별할 수 있어야 한다.

| 상황 | 앱의 처리 |
|---|---|
| 포트는 열렸는데 데이터 0건 | 'RX 0건' 카운터 + '수신없음' 상태 → 펌웨어 출력 문제로 좁힘 |
| 데이터가 깨져서 들어옴 | 파싱 실패로 분류, **원문 그대로 로그 표시** → 포맷/Baudrate 불일치 진단 |
| 값이 비정상적으로 튐 | 원시값을 그대로 표시(보정 안 함) + 범위초과 시 색상 경고 → 보드 측 문제 판별 |
| 케이블 분리 | 즉시 '끊김' 상태 + 시각 기록, 재연결 시 복구 안내 |
| 포트를 다른 프로그램이 점유 | 연결 실패 사유를 로그에 명시(예: 'Access denied') |
| 각도 미지원 펌웨어 | 거리만 표시, 각도 'N/A' (앱이 죽지 않음) |
| **거리값이 안 나옴 (세션 미점화)** | 세션 타임라인을 보고 단계 구분: `BLE_ADV`에서 멈춤=폰이 안 깨어남, `BLE_CONN`까진 갔는데 `OOB_DONE` 못 감=파라미터/키 교환 실패, `RANGING`인데 DIST 없음=PHY 문제 |
| **`STATE:ERR` 수신** | 타임라인 해당 단계를 적색 + `REASON` 표시, 로그에 시각과 함께 기록 |
| 세션 메시지 미지원 펌웨어 | 타임라인 'N/A' 처리, 거리 표시 기능은 그대로 동작 |

## 8. 검수 기준 (Acceptance Criteria)

아래가 모두 충족되면 "초도 테스트 도구로 사용 가능"으로 판정한다.

1. 보드 연결 후 **5초 이내**에 거리값이 레이더·수치 패널에 표시된다.
2. 태그를 좌/우/원근으로 움직이면 점이 **각도·거리에 맞게** 이동한다. (각도 지원 시)
3. 케이블을 뽑으면 **2초 이내** '끊김/수신없음'으로 상태가 바뀐다.
4. 깨진 데이터를 흘려도 앱이 **죽지 않고** 로그에 원문이 남는다.
5. 로그를 통해 '데이터 없음'과 '파싱 실패'를 **구분**할 수 있다.
6. **(OOB 관찰)** 세션 타임라인이 `BLE_ADV → BLE_CONN → OOB_DONE → RANGING` 진행을 단계별로 표시하고, 거리값이 없을 때 **어느 단계에서 멈췄는지** 식별된다.
7. **(OOB 실패)** `STATE:ERR,REASON:...` 수신 시 해당 단계가 사유와 함께 적색 표시된다.
8. (선택) 로그를 파일로 저장해 사후 확인할 수 있다.

### 9.3 데이터 계약 (코드가 따를 단일 진실)

```python
# parser.parse_line("DIST:85,ANGLE:-12") ->
{
  "dist_cm": 85,        # int | None
  "angle_deg": -12,     # int | None  (없으면 None = N/A)
  "raw": "DIST:85,ANGLE:-12",
  "ok": True,           # 파싱 성공 여부
  "error": None,        # 실패 시 사유 문자열
  "ts": 1750000000.21   # 수신 시각(epoch)
}
```

- 키 순서 무관, 대소문자 무관. `DIST`만 있어도 성공 처리(angle=None).
- 숫자 변환 실패/빈 줄 → `ok=False`, `error`에 사유, `raw` 보존.
- 범위(거리 0~5000cm, 각도 −90~+90)를 벗어나면 `ok=True`지만 UI에서 경고색 표시.

`STATE:` 로 시작하는 라인은 **세션 이벤트**로 분기 파싱한다:

```python
# parser.parse_line("STATE:OOB_DONE") ->
{ "kind": "state", "state": "OOB_DONE", "reason": None, "raw": "...", "ts": ... }
# parser.parse_line("STATE:ERR,REASON:OOB_TIMEOUT") ->
{ "kind": "state", "state": "ERR", "reason": "OOB_TIMEOUT", "raw": "...", "ts": ... }
# 측정 라인은 kind="measurement" 로 구분
```
- `state` 값: SLEEP / BLE_ADV / BLE_CONN / OOB_DONE / RANGING / ERR.
- 앱은 `measurement`는 레이더·수치 패널로, `state`는 세션 타임라인으로 라우팅한다.

### 9.4 동시성 모델 (필수 규칙)

- 시리얼 읽기는 **반드시 백그라운드 스레드**에서 `readline()` 블로킹 루프.
- 수신 결과는 `queue.Queue`에 넣고, **메인(UI) 스레드**가 주기적으로(예: 50ms) 꺼내 화면 갱신.
- UI 위젯은 메인 스레드에서만 갱신(`page.update()`), 스레드에서 직접 위젯 건드리지 말 것.
- 연결 해제 시 스레드 안전 종료(플래그 + `serial.close()`), 좀비 스레드 금지.

### 9.5 레이더 렌더 규칙 (6.2 재확인 — 코드용)

```
중심(cx, cy) = 캔버스 중앙
r_px = (dist_cm / max_dist_cm) * radar_radius_px      # 거리→픽셀
θ = radians(angle_deg)
x = cx + r_px * sin(θ)
y = cy - r_px * cos(θ)        # 0°가 화면 위(12시)로 가도록
```
- 동심원 눈금: 50/100/200/300cm (max_dist 기본 300cm, 설정 가능).
- 각도 가이드선: −90/−45/0/+45/+90°.
- 데이터 없음·angle=None이면 점 미표시(거리만 있으면 0° 가정 표시 옵션).

### 9.6 구현 우선순위 (이 순서로 만들 것)
1. 시리얼 연결 + 로그 콘솔에 원시 데이터 흘리기 (FR-1, FR-5) ← **여기까지만 돼도 1차 검증 가능**
2. 파서 + 수치 패널 (FR-2, FR-4)
3. 360° 레이더 캔버스 (FR-3)
4. 상태 표시·예외 처리 (FR-7, 7장)
5. 로그 저장·명령 전송 등 선택 기능 (FR-6, FR-8)

### 9.7 개발 중 테스트 팁 (보드 없이)
- 실제 보드가 없을 때를 대비해, `DIST:xx,ANGLE:yy`를 가짜로 흘려보내는 **시뮬레이터 모드/스크립트**(랜덤 워크 값)를 함께 만들면 UI를 보드 없이 검증 가능. (`com0com` 가상 포트 또는 내부 큐 주입)
